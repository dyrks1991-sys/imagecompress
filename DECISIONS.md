## ADR-001 — Use browser-image-compression over raw Canvas API

**Status:** Accepted
**Date:** 2026-07-06

### Context

Image compression can be done via canvas.toBlob() directly. However, JPEG quality via canvas is inconsistent across browsers and PNG output is always larger.

### Decision

Use browser-image-compression v2 which uses WASM encoders for consistent, high-quality compression across all browsers.

### Consequences

Adds ~150KB to bundle. Provides consistent output. Runs in Web Worker automatically.

## ADR-002 — Single-page architecture (no routing)

**Status:** Accepted
**Date:** 2026-07-06

### Context

The product is a single tool. Multiple pages would add navigation complexity and reduce perceived speed.

### Decision

Single page (/) only. Static export. No client-side routing needed.

### Consequences

Simplest possible architecture. Easy to expand to multi-page if needed in v0.2.

## ADR-003 — React state only — no Zustand or Redux

**Status:** Accepted
**Date:** 2026-07-06

### Context

State is a linear pipeline: file → compress → preview → download. No cross-component async complexity.

### Decision

useState in page.tsx for all state. Pass down as props. No global store.

### Consequences

Simple, fast. If batch processing (v0.2) requires complex state, migrate to useReducer at that point.

## ADR-004 — Object URL lifecycle management

**Status:** Accepted
**Date:** 2026-07-06

### Context

createObjectURL leaks memory if URLs are not revoked. This was a bug pattern identified in QuickQR development.

### Decision

Revoke Object URLs in useEffect cleanup. Never store the Blob — only the URL. Follow QuickQR pattern.

### Consequences

Zero memory leaks. Pattern documented in src/lib/download.ts.
